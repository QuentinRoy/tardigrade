| --- | --- | --- |
| Project | Grid | table, `id`/`row_id`, routes (`/projects` → `/grids`), cache tags, code identifiers, UI |
| Question | Rubric | table, FKs, code, UI, YAML import (`questions:` → `rubrics:`) |
| Rubric (leaf) | Criterion | tables (`*_rubric*`), FKs, code, UI, CSV columns |
| Team | Group | table, FKs, code, UI, students CSV (`team` → `group`) |
| Submission | Grade Target | table, FKs, `id`/`row_id`, code; **not** user-facing (UI/URLs name the Student or Group; `[targetId]` param only) |
| Assessment (record + act) | Grade | tables (`assessment`, `rubric_assessment`, `*_rubric_assessment`), `saveAssessment`, code, UI |
| Assessment Completion | Grade Completion | code, projections |
| Submission Matrix | Grade Matrix (internal) / "Grades" (UI) | component, aria-label, UI |
| Rubric Analytics | Criterion Analytics (internal) / "Analytics" (section on Results) | component, UI |
| Rubric overview (page) | **Results** page (Grades + Analytics); grid home "Dashboard" → **Overview** | page titles, routes, nav |
| Marks (aggregate) | Total | export columns (`grand_total_marks` → `final_total`), code |
| progress (as a synonym for completion) | Completion | `progress`/`progressPromise`/`progressLabel` props, `CompletionProgress` component, UI copy — unify on Completion (the concept `CONTEXT.md` settled) |
| boolean / ordinal / numerical (criterion kinds) | Check / Options / Number | `criterion_kind` DB enum (was `rubric_type`), subtype tables (`boolean_rubric*` → `check_criterion*` etc.), code, YAML `kind:` field + values, UI kind selector — one vocabulary end to end, no internal/external split. The classifier word is `kind`, matching the grade row's `kind` column (never "type"). **Folded into stage 2a** (shares the leaf rename's subtype-table migration — one hop, tables touched once). |

Value pipeline is **Grade → Mark → Total** for every criterion kind; a grade's recorded content varies by kind (a Yes/No answer, a label, or — for Number criteria only — a value). `grade` is the record and the act, distinct from its worth (a Number grade is itself a number, but never the marks value); `value` (never "score") names a Number grade's payload and the input axis of its configuration (`minValue..maxValue`), not a pipeline stage. `Points` remains avoided.

## Staging

Each stage is one or more PRs, own migration where schema changes, Kysely types regenerated, tests updated. Order minimizes cross-stage churn (rename leaf-inward and identifier-outward). Per `docs/reference/database-migrations.md`, committed migrations are not rewritten — every schema rename is a new migration.

**Sweep scope — include `e2e/`.** Any rename sweep must cover `e2e/` (specs and `e2e/fixtures/`), not just `src/` and `app/`. The e2e specs drive the real UI: they match on-screen labels (e.g. the import form's `Rubrics YAML` field, the `Rubrics assessed` completion tile), navigate literal route paths (`/import/rubrics`), and load fixture files whose top-level keys are contract-shaped (`rubrics:`). A rename that skips `e2e/` compiles and passes unit/integration but breaks `test:e2e` with stale-label timeouts, and fixture files can silently drift from the wire format. Stage 2b hit exactly this — `e2e/fixtures/questions.yaml` and the spec's `Questions YAML`/`/import/questions`/`Questions assessed` were missed on the first pass. Also watch **ALL-CAPS constants** (`QUESTIONS_YAML_PLACEHOLDER`): a case-sensitive `s/Question/Rubric/; s/question/rubric/` pass won't catch `QUESTIONS`, so add an explicit all-caps rule or grep for the upper-cased term afterward.

**Validation per stage — run the full gate, including `test:e2e`.** The complete gate is `check --fix`, `check-types`, `lint:boundaries`, the full `vitest` suite (unit + integration + storybook), production `build`, **and `test:e2e`** (it spins up its own ephemeral Postgres + built app and exercises import → grade → persist → export end-to-end — the only check that catches stale UI labels, routes, and fixture drift). `test:e2e` is not optional for these sweeps; a green unit/integration run is not sufficient. Note the storybook project can flake under the combined three-project `vitest` run (runner contention); re-run `test:storybook` in isolation to confirm before treating a storybook failure as a real regression.
