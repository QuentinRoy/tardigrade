# Criterion-kind vertical modules migration

- **Status:** Active
- **Created:** 2026-07-16
- **Origin:** [ADR 0013](../docs/adr/0013-criterion-kind-vertical-modules.md); [criterion-kind ownership investigation](../docs/investigations/2026-07-16-criterion-kind-ownership-and-persistence.md)
- **Tracked by:** #273

## Purpose

Execute [ADR 0013](../docs/adr/0013-criterion-kind-vertical-modules.md): make each criterion kind a vertical module under `src/criteria/{check,options,number}/`. Every step is **behavior-preserving** — file moves, seam extraction, and type collapse only; no product behavior changes, and `main` never goes red. Read ADR 0013 for the decision and the investigation for the rationale and rejected alternatives; this plan owns *how* and *in what order*.

## Strategy: pilot one kind, then replicate

Convert **one kind end-to-end first**, standing up the generic scaffolding it needs, and land it as one green PR. The other two kinds then follow the proven template, one PR each.

**Pilot: Check** — the simplest kind (true/false marks; no ranges, interpolation, or reconciliation). Piloting the simplest kind isolates the *structural machinery* risk (folder shape, coordinators, client/server split, dependency-cruiser interplay) from domain complexity. Then **Number** (stress-tests holding complexity; collapses the four-plus bounds-invariant copies), then **Options** last (its stale-mark reconciliation is the messiest persistence).

### PR2 — Number — **In review ([#298](https://github.com/QuentinRoy/tardigrade/pull/298))**

Replicate the template for Number; collapse the four-plus bounds-invariant copies into `criteria/number/`'s invariant consumed by editor schema, import schema, grade-save, and marking.

**As-built deviations (all behavior-preserving):**

- **The bounds invariant lives in its own dependency-free `criteria/number/numberBounds.ts`, not inside `numberDomain.ts`.** `numberSchemas.ts` (the editor + import leaves) must consume the predicates, while `numberDomain.ts` imports `NumberCriterionEditorValue` type-only from `numberSchemas.ts` (the zero-zod `createNumberCriterion` factory, per PR1's client-bundle fix). Keeping the predicates in `numberDomain.ts` would form a `numberDomain ↔ numberSchemas` cycle that dependency-cruiser rejects (it tracks the type-only edge). `numberBounds.ts` is the single invariant owner (`isNumberValueRangeValid`, `isNumberMarksRangeValid`) consumed by the editor schema, import schema, and grade-save.
- **Marking does NOT consume the strict bounds predicate.** Per CONTEXT.md **Number Criterion Bounds**, `markNumberCriterion` tolerates an inverted range and guards only the zero-width case, a deliberately weaker rule than `minValue < maxValue`. So the "four-plus copies" collapse covers editor/import/grade-save (the DB CHECK stays SQL); marking keeps its own tolerant zero-width guard.
- **Grade-save extracts the subtype write, not the validation orchestration.** `writeNumberGradeInDb` moved into `criteria/number/` (mirrors PR1's `writeCheckGradeInDb`); the value-range check now consumes the shared predicate, but the grade-validation flow + user-facing messages stay in `grade-persistence/gradeMutations.ts`. Fuller grade-validation relocation is deferred — PR1 didn't move Check's validation either (Check had none), so the coordinator's monolithic per-kind savers are untouched here.
- **The editor bounds refine moved onto the number editor leaf (`numberCriterionEditorSchema.superRefine`), not the rubric-level `superRefine`.** This is exactly PR1's acceptance criterion (per-kind refines move into the kind schema with relative paths `["maxValue"]`/`["maxMarks"]`, so composed `criteria[i].*` paths are unchanged) and mirrors the import leaf, which already self-validates its bounds. One benign consequence of leaf placement: in a rubric where one criterion fails base parse **and** a sibling is out-of-bounds, both issues now surface, whereas the old rubric-level `superRefine` was skipped entirely when any array element failed base parse (so it emitted only the base-parse issue). Issue paths and messages are unchanged; the change only makes multi-error feedback more complete, and no test exercises the mixed case.
- **The Number grade control stays flat in `criteria/` (`NumberGradeControl.tsx`), not moved into `criteria/number/`.** This matches the Check/Options grade controls, which PR1 also left in place ("grade control already present"); ADR 0013 lists grade controls as folder-owned, so relocating all three together is deferred (candidate for PR4). Only the authoring **fields fragment** (`NumberEditorFields.tsx`) and the **details projection** (`describeNumber`) moved into the folder in PR2.
- **`toNumberCriterion` drops the redundant `toNumber()` coercion** the old `rubrics.ts` number branch applied to the four `numeric` fields. Verified safe and matches the Check mapper (`toCheckCriterion` likewise trusts numbers): `src/db/kysely.ts` installs a process-wide `pg` type parser (OID 1700 → `Number`), so numeric columns already arrive as `number`; the old `toNumber` was a defensive no-op. The still-inline Options branch keeps its `toNumber` until PR3.
- **Number YAML encode is now an explicit `encodeNumberCriterion` projection** (was pass-through through the dispatcher). Test snapshots preserved; production number-criterion YAML key order is now normalized (semantically identical, untested) — same as PR1's check-encode note.
- **`CriterionSubtypeInput`'s number variant now derives from `NumberCriterionEditorValue` (`z.output`)**, removing the coordinator's transient dependence on the domain union for Number (as the plan foresaw); the Options variant still derives from the domain union until PR3.
- **Test-seam migration** matched PR1: `criterion.test.ts` was **renamed** to `criteria/number/numberDomain.test.ts` (git tracks it as a rename; it carried the `markNumberCriterion` cases, which were the file's only content) and expanded with `describeNumber`/`exportNumberGradeValue`/`encodeNumberCriterion` cases; added `numberBounds.test.ts` (invariant) and `numberPersistence.integration.test.ts` (batched upsert + coordinator dispatch, using the `inTransaction` helper). The Number cases in `schemas.test.ts`, `parseRubrics.test.ts`, `rubrics.test.ts`, `resultsBuilder.test.ts`, and `rubricsExport.test.ts` were **left in place** — they stay green through the new seams; folded into the PR4 test revision.
- **Rebased onto [#297](https://github.com/QuentinRoy/tardigrade/pull/297)** (write primitives typed `Transaction<Database>`, ADR 0007): the Number persistence adapters take `Transaction<Database>` and the integration test uses the new `inTransaction` test helper, matching the updated Check adapters.

#### PR4b — Test revision — **In review ([#301](https://github.com/QuentinRoy/tardigrade/pull/301))**
